El Límite de las Tablas 🧱

Hasta ahora, todo nuestro mundo han sido las tablas. Pero en un sistema real, una tabla por sí sola tiene problemas de seguridad y eficiencia. Pasa el cursor sobre cada tarjeta.

El Problema de la Privacidad

Si le das a alguien acceso a la tabla "Empleados", podrá ver TODAS sus columnas. ¿Qué pasa si quieres que vean los nombres pero NO los salarios? Necesitamos Vistas.

El Problema de los IDs

Insertar IDs manualmente (1, 2, 3...) es lento y propenso a errores en sistemas con miles de usuarios concurrentes. Necesitamos un generador automático: Secuencias.

El Problema de "Soy Dios"

Un sistema no puede tener un solo usuario administrador para todo el equipo. Necesitamos crear cuentas separadas y limitar qué pueden hacer: Control de Acceso.

Arquitectura Real

Una Base de Datos no es solo una hoja de cálculo gigante. Es un motor complejo que maneja seguridad, automatización y múltiples capas lógicas para proteger la información.

Vistas (Views): Las Tablas Virtuales

Una Vista es una consulta SELECT guardada en la base de datos con un nombre. Se comporta exactamente igual que una tabla cuando la consultas, pero no almacena datos físicos por sí sola; los extrae de las tablas originales en tiempo real.

Para Seguridad

Oculta columnas sensibles (como salarios o contraseñas). Le das acceso al usuario a la Vista, en lugar de a la Tabla original completa.

Para Simplicidad

Si tienes un JOIN súper complejo que usas todos los días, guárdalo como una Vista. Luego solo haces SELECT * FROM tu_vista.

-- 1. Creamos la vista para ocultar el salario
CREATE OR REPLACE VIEW vista_empleados_publico AS
SELECT id, nombre, cargo 
FROM empleados; -- Nota: NO seleccionamos la columna 'salario'

-- 2. El usuario consulta la vista como si fuera una tabla
SELECT * FROM vista_empleados_publico;

Secuencias: Generadores Numéricos

Una Secuencia es un objeto que genera números secuenciales y únicos (1, 2, 3...). Son independientes de las tablas y se utilizan casi exclusivamente para rellenar de forma automática las Claves Primarias (Primary Keys) de las tablas al hacer un INSERT.

El Motor de la Secuencia

La magia ocurre con dos palabras clave que debes añadirle al nombre de tu secuencia:

  • .NEXTVAL: Genera el siguiente número en la secuencia y lo devuelve. (Lo usas para insertar).
  • .CURRVAL: Devuelve el número actual que generaste en tu sesión, sin incrementarlo. (Útil para revisar en qué número vas).
-- 1. Crear la secuencia
CREATE SEQUENCE seq_empleados 
START WITH 100 
INCREMENT BY 1;

-- 2. Usar la secuencia en un INSERT
INSERT INTO empleados (id, nombre, cargo, salario) 
VALUES (seq_empleados.NEXTVAL, 'Ana', 'Dev', 2000); 
-- Ana obtiene el ID 100

INSERT INTO empleados (id, nombre, cargo, salario) 
VALUES (seq_empleados.NEXTVAL, 'Beto', 'Ops', 1800); 
-- Beto obtiene el ID 101

DCL: Control de Acceso y Privilegios

No todos pueden hacer todo. El Data Control Language (DCL) nos permite administrar la seguridad de nuestra base de datos creando usuarios y otorgando (o quitando) permisos específicos mediante los comandos GRANT y REVOKE.

Privilegios de Sistema

Permisos generales sobre la base de datos entera. Ej: Poder conectarse, poder crear tablas, crear usuarios.

GRANT CREATE SESSION TO juan;

Privilegios de Objeto

Permisos sobre un objeto específico (una tabla o vista). Ej: Poder hacer SELECT o UPDATE solo a la tabla 'empleados'.

GRANT SELECT ON empleados TO juan; 
-- 1. Crear el usuario (requieres ser administrador)
CREATE USER analista_rrhh IDENTIFIED BY "Password123";

-- 2. Darle permiso para que pueda iniciar sesión (Sistema)
GRANT CREATE SESSION TO analista_rrhh;

-- 3. Darle permiso para ver nuestra VISTA segura (Objeto)
GRANT SELECT ON vista_empleados_publico TO analista_rrhh;

-- Si se porta mal, le quitamos el permiso
REVOKE SELECT ON vista_empleados_publico FROM analista_rrhh;

Pregunta Frecuente: ¿Por qué crear usuarios en la BD si existe el "Frontend"?

Muchos estudiantes se preguntan: "¿Para qué crear un usuario us_cobranzas en la base de datos si el analista de cobranzas va a usar una página web (Frontend) o una app, y no va a ejecutar código SQL manualmente en la consola?"

La respuesta está en la Arquitectura Segura:

  • El Frontend se comunica con una API (Backend). Es el código de esa API el que hace la conexión real a la base de datos.
  • Si el Backend se conecta a Oracle usando un superusuario administrador, y un hacker logra encontrar una vulnerabilidad en la web, ¡tendrá control total de todos tus datos!
  • Para evitar esto, el Backend se configura para que se conecte usando credenciales limitadas (ej: nuestro us_cobranzas). Así, la seguridad está garantizada a nivel de motor. Si la web es hackeada, el atacante igual no podrá ver los diagnósticos de los pacientes, porque el motor de base de datos le bloqueará el paso (Principio de Menor Privilegio).

Cheatsheet: Comandos DCL y Variantes

Comando / Variante ¿Qué hace? Ejemplo Práctico
CREATE USER Crea una nueva cuenta de usuario y le asigna una contraseña. CREATE USER pablo IDENTIFIED BY "p123";
GRANT CREATE SESSION Da permiso para iniciar sesión (conectarse a la BD). Esencial, sin esto el usuario no sirve. GRANT CREATE SESSION TO pablo;
GRANT SELECT Permite leer (hacer SELECT) datos de una tabla o vista específica. GRANT SELECT ON empleados TO pablo;
GRANT INSERT, UPDATE, DELETE Permite modificar datos. Puedes otorgar varios separándolos por comas. GRANT INSERT, UPDATE ON empleados TO pablo;
GRANT ALL PRIVILEGES Da TODOS los permisos sobre un objeto específico. Útil para administradores. GRANT ALL PRIVILEGES ON empleados TO lider_db;
REVOKE Quita un permiso previamente otorgado. Su sintaxis usa FROM en lugar de TO. REVOKE DELETE ON empleados FROM pablo;

Demostración en Vivo: Arquitectura Completa

Copia este script y ejecútalo para ver la sinergia entre tablas, secuencias, vistas y permisos.

script_demo_arquitectura.sql 
-- (Si ya creaste la tabla 'empleados' en la preparación, saltate este paso)
-- 1. Aseguramos la tabla base
CREATE TABLE hr_empleados (
    id NUMBER PRIMARY KEY,
    nombre VARCHAR2(50),
    salario NUMBER
);

-- 2. Creamos el Generador Automático (Secuencia)
CREATE SEQUENCE seq_hr_emp START WITH 1 INCREMENT BY 1;

-- 3. Insertamos usando la Secuencia en lugar de escribir los IDs a mano
INSERT INTO hr_empleados (id, nombre, salario) VALUES (seq_hr_emp.NEXTVAL, 'Carlos', 2500000);
INSERT INTO hr_empleados (id, nombre, salario) VALUES (seq_hr_emp.NEXTVAL, 'Diana', 1800000);
COMMIT;

-- 4. Creamos la VISTA segura (sin el salario)
CREATE OR REPLACE VIEW vista_hr_publico AS
SELECT id, nombre FROM hr_empleados;

-- 5. Demostración de Control de Acceso (Requiere permisos de Admin)
-- Creamos un usuario de solo lectura
-- (Nota: Dependiendo de tu entorno puede que no tengas permisos para crear usuarios. 
-- Si da error, ignora los pasos de GRANT/CREATE USER)
CREATE USER consultor_externo IDENTIFIED BY "Ext123";
GRANT CREATE SESSION TO consultor_externo;

-- Le damos acceso SOLO a la vista segura, NO a la tabla base
GRANT SELECT ON vista_hr_publico TO consultor_externo;

-- Si nos loguearamos como 'consultor_externo' e hiciéramos:
-- SELECT * FROM hr_empleados; --> DARÍA ERROR DE PERMISOS
-- SELECT * FROM vista_hr_publico; --> FUNCIONARÍA y solo vería (id, nombre)

Cultura Dev: El Principio de Menor Privilegio

"Every program and every privileged user of the system should operate using the least amount of privilege necessary to complete the job."

— Jerome Saltzer & Michael Schroeder (1974)

El Origen en el MIT

Lo que hoy te suena como algo "obvio" (no le des la contraseña de administrador al pasante), en realidad fue formalizado como un principio arquitectónico en 1974 por investigadores del MIT mientras desarrollaban Multics (el abuelo de Linux). Nació porque en los primeros computadores compartidos, si un programa fallaba, destruía el sistema entero. Necesitaban "contenedores" de permisos.

La Definición Formal

El Principio de Menor Privilegio (PoLP) dicta que un usuario o programa debe recibir únicamente los privilegios estrictamente necesarios para realizar su tarea, y por el tiempo mínimo requerido. Ni un permiso más.

Por qué salva empresas hoy

Si un Ransomware (virus de secuestro) entra al computador de alguien, el virus "hereda" sus permisos. Si aplicaste PoLP, el virus no podrá destruir la base de datos, bloqueando lo que en ciberseguridad se llama "movimiento lateral".

Evolución: Zero Trust y Cloud Computing

Hoy en día, este principio de 1974 es la base absoluta de la Nube (AWS, Google Cloud, Azure) y de la moderna arquitectura Zero Trust (Confianza Cero). En la nube ya no existen contraseñas compartidas para las aplicaciones; se usan roles ultra-específicos (IAM).

Aprender a usar GRANT y REVOKE no es solo "una tarea de Base de Datos", es entrenar tu mente para pensar como un Cloud Architect o un ingeniero de Ciberseguridad.

IDENTITY vs SEQUENCE

Durante décadas, en Oracle la única forma de hacer una clave primaria autoincremental era creando una SEQUENCE y usando un Trigger (o llamando a .NEXTVAL en los INSERTs). Sin embargo, a partir de Oracle 12c (2013), se introdujo GENERATED ALWAYS AS IDENTITY para acercarse a lo que otros motores (como SQL Server o MySQL) ya hacían.

La Forma Clásica (Sequences)

  • Requiere dos objetos: la tabla y la secuencia.
  • Debes escribir .NEXTVAL en cada INSERT explícitamente.
  • Súper Flexible: Una misma secuencia puede alimentar múltiples tablas.
  • Permite extraer números antes de insertar (ej: para usarlos en transacciones relacionadas).

La Forma Moderna (Identity)

id NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY
  • Más fácil de escribir. El motor hace todo por detrás.
  • Código más limpio en los INSERTs (solo pasas las otras columnas).
  • Si intentas insertar un ID manualmente, Oracle lanzará un error si usaste "ALWAYS".
  • Menos control manual si necesitas manipular el número antes de guardarlo.

¿Qué se usa en la industria real?

Aunque IDENTITY es mucho más moderno y cómodo, en la banca, retail y grandes corporativos verás que casi todos usan Sequences. ¿Por qué?

1. Sistemas Legacy (Antiguos): Gran parte de la infraestructura crítica financiera se escribió antes de 2013 (Oracle 11g o inferior). Las secuencias eran la única opción, y reescribir la base de datos de un banco solo por usar una sintaxis nueva no es rentable ni seguro.

2. Arquitecturas ORM (Hibernate / Java): Herramientas empresariales gigantes en Java muchas veces están configuradas para pedir el número a la base de datos ANTES de insertar (SELECT seq.NEXTVAL). Usan ese número para amarrar otras lógicas en memoria antes de siquiera mandar el INSERT real a la base de datos.

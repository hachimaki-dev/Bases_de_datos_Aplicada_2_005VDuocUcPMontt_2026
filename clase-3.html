Momento 0: Recordando nuestras Raíces (Normalización)

El Peligro del "Mega-Excel"

Imaginen por un segundo que Netflix guardara su información en una tabla gigante y plana. Cada vez que tú vieras un episodio de *Stranger Things*, ¡tendrían que volver a escribir tu nombre completo, tu email, los datos de tu tarjeta de crédito, el nombre de la serie, el director y el género!

Ustedes ya saben cómo solucionar esto: ¡Normalizar! Mediante las Formas Normales (1FN, 2FN, 3FN) dividimos esa monstruosidad en entidades limpias (`Usuarios` por un lado, `Contenidos` por otro) y las enlazamos usando Llaves Foráneas (FK) con un diagrama Entidad-Relación (DER).

💡 La Verdad sobre el JOIN

Destruimos la información monolítica mediante la Normalización para no redundar datos en el disco duro. Pero ahora, cuando el gerente quiere ver un reporte entendible por humanos... ¡necesitamos volver a juntarlos de manera temporal en la RAM! El JOIN es el costurero mágico que vuelve a unir las piezas del rompecabezas.

⭐ Meta-reflexión sobre Cardinalidad (1:1, 1:N, N:M)

¿Te diste cuenta? Un usuario tiene muchas compras (1:N), pero un Ticket tiene muchos productos y un Producto está en muchos Tickets (N:M). Para armar esa boleta, SQL tuvo que hacer un JOIN gigante usando llaves foráneas para cruzar tres entidades separadas en el DER gracias a una tabla "Detalle de Compra".

Momento 2: Construyendo la Query paso a paso

Cuando quieras hacer un JOIN de 3 o más tablas partiendo de tu DER, NUNCA lo escribas todo de una vez. Se construye progresivamente, parándonos visualmente en la tabla central (N) y "viajando" hacia las maestras (1).

1. La base y el Alias

Primero nombramos la tabla puente y le damos un "alias" o apodo cortito (`v`) para no escribir la palabra gigante a cada rato.

SELECT *
FROM visualizaciones v;

2. El Primer Puente (Hacia Usuarios)

Acoplamos la maestra usuarios u. El ON es simplemente la declaración de igualdad entre la Foránea y la Primaria.

SELECT *
FROM visualizaciones v
JOIN usuarios u 
  ON v.id_usuario = u.id_usuario;

3. El Segundo Puente (Hacia Contenidos)

Finalizamos el diseño. Sutilmente dejamos de usar el arterisco (`*`) y elegimos campos legibles gracias a los alias.

SELECT u.nombre, c.titulo, v.progreso
FROM visualizaciones v
JOIN usuarios u ON v.id_usuario = u.id_usuario
JOIN contenidos c ON v.id_contenido = c.id_contenido;

Momento 3: El Valle de los Errores

El motor relacional hará literalmente lo que tú le digas, incluso si eso significa arruinar el servidor o entregar métricas falsas que quiebren la compañía. Analiza estos 2 desastres muy comunes:

ERROR LÓGICO: Claves Destruidas

Un estudiante escribió mal la condición del ON, cruzando la Edad o el ID erróneo.

SELECT u.nombre, v.progreso
FROM usuarios u
JOIN visualizaciones v 
  ON u.id_usuario = v.id_contenido; -- ¡AUCH!

Consecuencia Práctica: SQL no dará error (¡ambos son solo números TYPE NUMBER de Oracle!). Pero unirá al "usuario 5" con "la película 5". Generarás un reporte donde un joven compró cosas que no tienen sentido solo porque la ID del usuario coincidió mágicamente con la PK que no era.

ERROR FATAL: El Producto Cartesiano

Un estudiante olvidó escribir la condición matemática `ON` que acopla las entidades.

SELECT u.nombre, c.titulo
FROM usuarios u
JOIN contenidos c; -- !FALTA EL ON!

Consecuencia Operativa: Al no tener una llave (PK=FK) para guiarse, la Base de Datos asume que quieres cruzar TODOS contra TODOS indiscriminadamente.

Si tu web tiene 10.000 usuarios y 50.000 películas... el resultado generará 500 millones de filas inservibles (10.000 * 50.000) colapsando la CPU, llenando la memoria e interrumpiendo las compras reales del e-commerce.

SQL asume que quieres emparejarlos absolutamente todos a la fuerza matemáticamente (2 artistas × 3 comidas = 6 filas). El servidor no discriminará y te devolverá este monstruo que no respeta la realidad:

Desafío de Máquinas: Congela tu Propio Computador

Momento 5: Escalando el Pensamiento

El Misterio de la Usuaria Fantasma

Imagina una usuaria VIP que pagó su suscripción pero durante 3 meses, jamás presionó 'Play' en ninguna serie. No hay ningún registro de ella en la tabla N:M puente `visualizaciones`.

Si corremos tu exitoso reporte corporativo con `INNER JOIN`... ¡Esta usuaria desaparecerá del Excel final como si no existiera!

El `INNER JOIN` exige 100% de coincidencias y reciprocidad. Discútelo con clase: ¿Qué tipo de reporte estadístico necesitaría ver el gerente, donde es **obligatorio** mostrar clientes incluso cuando tengan `NULL` o `CERO` compras?

Prepárate... para resolver esto usaremos pronto la Fuerza de Gravedad: El LEFT OUTER JOIN.

-- Script Oracle SQL: Entorno Netflix Básico
DROP TABLE visualizaciones;
DROP TABLE contenidos;
DROP TABLE usuarios;

CREATE TABLE usuarios (
    id_usuario NUMBER PRIMARY KEY,
    nombre VARCHAR2(50),
    pais VARCHAR2(50),
    fecha_registro DATE
);
CREATE TABLE contenidos (
    id_contenido NUMBER PRIMARY KEY,
    titulo VARCHAR2(100),
    genero VARCHAR2(50),
    duracion_min NUMBER,
    rating NUMBER(3,1)
);
CREATE TABLE visualizaciones (
    id_visualizacion NUMBER PRIMARY KEY,
    id_usuario NUMBER,
    id_contenido NUMBER,
    fecha DATE,
    progreso NUMBER,
    FOREIGN KEY (id_usuario) REFERENCES usuarios(id_usuario),
    FOREIGN KEY (id_contenido) REFERENCES contenidos(id_contenido)
);
